# Prism · 棱镜

把任何概念、判断、现象、产业逻辑或调研型理解问题变成一张可交互的 HTML 页面，让外行人也能看懂。

设计语言沿用 Kami：暖色羊皮纸画布 `#f5f4ed`、油墨蓝强调色 `#1B365D`、衬线字体主导层级。中文用 TsangerJinKai02，英文用 Charter。

---

## Step -1 · 触发边界与交付契约

Prism 不只用于单个名词。以下请求都必须触发：

- 用户要求解释一个**概念/术语/机制**：如“解释量子计算”“什么是 RAG”。
- 用户要求理解一个**判断/金句/观点**：如“为什么有人说短期缺芯片，长期缺能源，永远缺存储”。
- 用户要求拆解一个**现象/产业逻辑/商业判断**：如“AI 为什么会拉动电力和存储”“这个趋势背后的依据是什么”。
- 用户说“帮我调研/理解/讲清楚/解释一下/这里面的逻辑和依据/有哪些知识可学习”等，即使没有出现“概念”两个字。

交付契约：

- 默认最终产物是单个 `.html` 文件，内联 CSS/JS，包含右侧目录、阅读设置、SVG 图解和来源区。
- 最终回复只给文件路径、验证结果和少量说明；不要用纯文本正文替代 HTML 页面。
- 如果用户明确要求“直接在聊天里解释”，才可输出纯文本；否则 Prism 任务必须落到 HTML。
- 涉及时效性、产业、投资、政策、科学进展、产品能力或争议事实时，先联网检索或读取权威本地资料，再写页面。

## Step 0 · 调研与问题定型

如果用户的问题不是稳定常识，先做调研。调研不是为了堆资料，而是为了判断这句话/现象是否站得住。

### 调研型问题的最小来源要求

- 至少 3 类来源：原始说法或出处、权威/官方数据、产业参与者或研究机构材料。
- 优先级：官方报告/年报/论文/监管文件 > 公司公告 > 主流媒体 > 二手解读。
- 明确区分“事实依据”“合理推断”“叙事夸张”“投资结论”。
- 对每个核心判断写出反证或边界：什么时候这句话会失效，哪些环节可能被技术进步缓解。
- 在 HTML 末尾加入“依据与边界”或“来源” section，列出关键来源链接和它们支撑的结论。

### 调研到认知地图的转换

先把用户问题改写成一个可教学的目标：

```text
用户问题：为什么有人说短期缺芯片，长期缺能源，永远缺存储？
目标概念：AI 基建瓶颈会沿着算力 → 电力 → 数据底座迁移
读者最终要懂：这不是三句口号，而是资源约束在不同时间尺度上的排序
必须先懂：
1. AI 系统为什么先消耗算力
2. 算力扩张为什么会把瓶颈推给电力和机房
3. 数据为什么会持续累积，而不是像电一样被用完
4. 哪些证据支持这个判断
5. 哪些地方是叙事夸张或投资风险
```

然后按这个目标组织页面。不要把资料摘要堆成报告；要把资料变成读者能看懂的因果图、结构图、数据图和误区辨析。

## Step 1 · 判断讲解方式

**默认策略：通俗易懂，不假设用户任何领域背景。** 用日常生活场景做类比（快递、厨房、交通、游戏），所有专业术语首次出现时必须括号解释。

只有当用户**明确说出**自己的背景时才调整：

| 用户说了什么 | 怎么调 |
|---|---|
| "我是做XX的" / "我是XX行业的" | 用该行业做类比，跳过该行业基础概念 |
| "我是文科生" / "我没学过XX" | 避免该领域术语，加强生活类比 |
| 什么都没说（绝大多数情况） | **默认通俗模式**，不猜、不假设、不预设 |

## Step 2 · 分析概念，规划内容结构

收到概念后，按以下框架拆解：

### 先设计认知阶梯（复杂概念必须）

**默认不要把复杂概念直接拆成并列模块。** 先设计读者从零到懂的学习路径，尤其当概念涉及科学、工程、金融、密码学、AI、医学、法律等前置知识时。

生成内容前先写清：

```text
目标概念：量子计算
读者最终要懂：通过操控概率幅，让某些答案更容易被测到
必须先懂：
1. 普通 bit 是什么
2. 普通计算为什么会慢
3. 概率和测量是什么关系
4. 叠加不是读出所有答案
5. 干涉是加强/抵消概率幅
6. 量子门是改变状态的动作
7. 量子算法是一串安排好的动作
```

然后目录必须按这个阶梯展开。每一节只解决一个认知问题。不要在页面里写"下一节要讲什么"这类课程串场文案；递进关系应该通过目录顺序、章节标题和内容本身自然体现。

章节结构建议：

```text
本节只解决：为什么需要 X？
先补前置：不用术语讲一个生活场景
先用图/动画把生活场景跑通
再引入术语：刚才的 A 在专业里叫 X
最后收束本节：用一句话说明本节结论
```

如果读者理解某节需要前置概念，必须先补前置概念，不要用一句括号解释带过。

### 先场景后术语（外行模式硬规则）

面向纯外行时，不能一上来使用领域词作为解释入口。即使术语已经括号解释，也通常不够。

每个新术语第一次出现前，必须先完成三步：

1. **生活场景**：用读者已经懂的对象讲清动作和结果，例如白牌/蓝牌进入翻牌机。
2. **可视化演示**：用 SVG/CSS 动画把输入、规则、输出跑一遍，让读者先看到"发生了什么"。
3. **术语命名**：最后再说"刚才的白牌/蓝牌，在电脑里叫 0/1 bit；翻牌机，在电脑里叫 gate"。

禁止模式：`bit 是 0 或 1，NOT 门会把 0 变 1`。

推荐模式：`先看两张牌和一台翻牌机：白牌进去变蓝牌，蓝牌进去变白牌。电脑里把这两种确定状态写成 0 和 1，叫 bit；把翻牌机这种固定规则叫门。`

### 反直觉概念必须拆开讲

当概念明显违背日常经验时，不能只写"它能 X"。读者真正卡住的通常是"为什么它能 X"、"背后到底发生了什么"、"它是不是也能用在日常物体上"。

这类内容必须补齐 5 个问题：

1. **普通直觉**：先写读者会自然以为什么，例如"概率怎么能被改变？结果不是随机的吗？"
2. **机制入口**：用生活场景或可视化对象解释"被改变的到底是什么"，例如状态指针、波的相位、候选路径。
3. **现实承载物**：说明现实里由什么承载这个机制，例如电子、原子、离子、超导电路、激光/微波控制。
4. **适用边界**：说清它在哪些条件下成立，哪些对象或场景不能直接套用。
5. **常见关联**：如果读者可能想到著名例子或思想实验，必须解释关系和区别，例如薛定谔的猫与量子叠加/测量有关，但不是量子计算机真的用猫计算。

推荐结构：

```text
日常直觉为什么会困惑
先用可视化模型解释被改变的对象
再说现实硬件怎样施加控制
再说这个控制为什么会改变结果概率
最后说适用范围、边界和常见误区
```

### 必选模块（每个概念都要有）

1. **一句话定义** — 用最大字号展示，20 字以内说清本质
2. **生活类比** — 用一个日常场景做类比，配 SVG 示意图
3. **核心原理** — 用 SVG 动画/图解演示"它怎么工作的"
4. **关键组成部分** — 用 SVG 结构图拆解内部零件
5. **常见误区** — 至少 2 个"人们常搞错的地方"

### 可选模块（根据概念复杂度决定是否包含）

6. **历史背景** — 如果概念有有趣的历史，用时间线 SVG 展示
7. **与其他概念的关系** — 用 SVG 关系图展示它和相关概念的异同
8. **实际应用** — 用具体例子说明"生活中哪里用到了"
9. **进阶知识** — 如果用户水平较高，可以加"深入一层"的内容

### 结构规则

- 每个模块是一个 `<section>`，有独立的 `id` 用于目录跳转
- 模块之间用足够的留白分隔（72px margin-bottom）
- 每个模块开头有一行小字标注模块类型（如"核心原理 · How it works"）
- 复杂概念开头必须有"阅读路线/认知地图"：用 3-5 个步骤说明整篇文章的主线，让读者先知道为什么按这个顺序讲
- 简单概念可用：是什么 → 像什么 → 怎么工作 → 由什么组成 → 别搞错 → （可选）历史/关系/应用
- 复杂概念必须用递进学习线：为什么需要 → 普通基准 → 前置概念 → 核心直觉 → 机制分步 → 术语命名 → 系统组成 → 能与不能
- 不要过早抛出并列术语。比如解释量子计算时，不要一上来并列讲"量子比特、量子门、量子算法、纠错"；要先建立 bit、概率、测量、叠加、干涉这些台阶。
- 不要写显式串场提示，如"下一节为什么要讲..."、"现在你能理解下一节了..."。章节应当自洽，读者靠目录和视觉结构自然往下读。
- 模块之间要有文章式衔接：每个关键图解后用 1 个短段解释"这张图把哪个问题解决了，它和主线有什么关系"。不要写课程式"下一节..."，要写本节和主线的因果关系。

## Step 3 · 创建 SVG 图解

**这是本 skill 的核心能力。** 每个概念至少需要 3 个 SVG 图解。

### SVG 类型选择

#### 数据图表（35 种可用，详见 `references/proto-data-charts.html`）

| 要表达什么 | 推荐图表 | 候选 |
|---|---|---|
| 数值对比 | 柱状图、横向柱状图 | 分组柱状图、堆叠柱状图、象形图 |
| 变化趋势 | 折线图、面积图 | 面积堆叠图、迷你图、阶梯图 |
| 占比/构成 | 饼图、环形图 | 旭日图、树状图、漏斗图、玫瑰图 |
| 相关性/关系 | 散点图、气泡图 | 象限图、平行坐标图、维恩图 |
| 数据分布 | 直方图、箱线图 | 小提琴图、密度图、蜂群图 |
| 多维对比 | 雷达图 | 极坐标柱状图 |
| 流向/转化 | 桑基图、瀑布图 | 弦图 |
| 数值度量 | 仪表盘、进度条 | 热力图、K线图 |

#### 流程结构（35 种可用，详见 `references/proto-flow-structures.html`）

| 要表达什么 | 推荐结构 | 候选 |
|---|---|---|
| 步骤流程 | 线性流程、条件分支 | 循环流程、并行流程、交互式流程 |
| 状态/时序 | 状态机、时序图 | 活动图 |
| 系统设计 | 类图、组件图 | 用例图、ER 图、数据流图 |
| 数据/架构 | 数据管道、网络拓扑 | 部署图、泳道图 |
| 项目管理 | 甘特图、看板图 | 时间线 |
| 分析/决策 | 思维导图、决策树 | 鱼骨图、金字塔图、信息架构 |
| 组织/关系 | 组织架构图、依赖关系图 | 嵌套层级、事件链 |
| 用户体验 | 用户旅程图、故事板 | 服务蓝图 |
| 循环/反馈 | 循环图、反馈循环 | 飞轮效应图 |

#### 选择原则

- **默认**：每个概念至少选 3 种不同形式的 SVG（数据+流程+结构各一）
- **数据多**：优先用图表类（比较/趋势/构成/分布）
- **逻辑多**：优先用流程类（线性/分支/并行/状态机）
- **关系多**：优先用结构类（ER/类图/思维导图/金字塔）

### SVG 设计规范（沿用 Kami）

```
画布背景：#f5f4ed（parchment）
节点背景：#faf9f5（ivory）
强调色：#1B365D（ink blue）— 仅用于焦点节点
文字颜色：#141413（near-black）标题 / #504e49（olive）说明
边框色：#6b6a64（stone）常规 / #1B365D 焦点
箭头/连线：#504e49（olive）
字体：font-family: inherit（继承页面字体）
```

### SVG 动画规范

使用 CSS `@keyframes` 实现交互动画，**不依赖 JavaScript 库**：

```css
/* 常用动画 */
@keyframes fadeInUp { from { opacity:0; transform:translateY(20px); } to { opacity:1; transform:translateY(0); } }
@keyframes fadeInSvg { from { opacity:0; } to { opacity:1; } }
@keyframes draw { from { stroke-dashoffset: 1000; } to { stroke-dashoffset: 0; } }
@keyframes pulse { 0%,100% { opacity:1; } 50% { opacity:0.5; } }
@keyframes flow { from { offset-distance:0%; } to { offset-distance:100%; } }
@keyframes highlight { 0%,100% { fill:#faf9f5; } 50% { fill:#EEF2F7; } }

/* 交互触发 */
.svg-interactive:hover .node { /* 悬停高亮 */ }
.svg-interactive:active .step { /* 点击推进 */ }
```

**动画原则**：
- 动画用来**解释过程**，不是装饰
- 每个动画都有明确的教学目的（"看，数据从这里流到那里"）
- 提供 `prefers-reduced-motion` 媒体查询，尊重用户偏好
- 简单强调动画时长 0.5-2 秒；多阶段教学动画用 3-6 秒，让读者看清因果顺序
- 复杂概念优先做"阶段动画"：先出现对象 → 再出现作用力/流向/变化 → 最后高亮结论，不要所有元素同时循环
- **禁止孤立圆点/粒子沿隐形路径移动**。如果要表达流动，必须画出明确的管道、箭头轨道或阶段状态，并给流动对象命名；否则改用静态箭头、编号步骤或局部高亮。不要在一张图上放一个没有语义说明的点来回移动。
- **不要在带 `transform="translate(...)"` / `rotate(...)` 的 SVG 元素上使用会修改 CSS `transform` 的动画**（如 `fadeInUp`、`popIn`、`growUp`）。CSS 动画会覆盖 SVG 原本的坐标变换，导致文字、图例、图形错位。此类元素只能用 `fadeInSvg` 这类只改变 `opacity` 的动画，或改成外层负责透明度、内层负责 `transform` 的双层 `<g>` 结构。

### 专业级 SVG 教学动画规范

当用户要求"更专业"、"细节更好"、"符合物理规律"、"不要像 demo"，或概念本身涉及运动、波、力、流程、因果机制时，必须按教学动画组件来做。

#### 先写动画设计草图

在 SVG 代码附近用注释写清楚：

```html
<!-- animation intent:
  teaching claim: 同相作用会放大结果，反相作用会抵消结果
  visual mapping: 秋千摆幅 = 概率幅大小；推力箭头 = 相位相同/相反的作用
  sequence: 1 秋千运动 → 2 推力出现 → 3 摆幅变化 → 4 波形描线 → 5 结论高亮
  boundary: 秋千只是类比，量子中变化的是概率幅，不是物体在空间摆动
-->
```

如果无法用这 4 行讲清动画意图，先重构图解，不要直接写 SVG。

#### 物理或机制直觉

- 运动类比要符合基本直觉：摆动在两端慢、中间快；加速/减速要有方向原因；流体/数据流要沿明确通道移动。
- 箭头必须在正确时机出现：表示推力、因果、流向的箭头不能只是闪烁装饰，要与对象运动或状态变化同步。
- 波、振动、周期类图解要区分输入和合成结果：优先分成"波 A / 波 B / 合成"三行，不要把多条线重叠到读者猜不清。
- 概率、风险、强弱变化要用面积、振幅、亮度或柱高表达，不要只靠颜色深浅承载关键含义。
- 类比必须标注边界：用一句图注说明"这个类比只解释什么，不代表真实系统也这样运动"。

#### 图文分区与细节密度

- 专业动画优先使用更大的 viewBox，例如 `900 x 560`、`960 x 600`；不要把复杂动画塞进 `720 x 320`。
- 把画布分成独立区域：运动区、解释区、图例区、结论区。文字不能进入运动轨迹、箭头通道或波形主线。
- 每个区域内部留 24-32px 安全边距；标题和主体图形基线至少相隔 24px。
- 标签靠近线条、箭头、波形时必须加浅色底或放到独立标签带。
- SVG 内文字只放短标签和结论；超过 16 个中文字符的解释移到图注或正文。

#### 动画时序

- 教学动画要有因果顺序：对象出现 → 作用出现 → 状态变化 → 结果高亮。
- 多阶段循环建议 4-6 秒；读者应能在一轮循环里看出"为什么变了"。
- 不要让 5 个以上元素同时以同等强度运动；一次只突出一个教学重点。
- 使用 `prefers-reduced-motion` 时，静态画面仍必须能说明关系，不能依赖动画后才看懂。

#### 专业动画质量门槛

完成后必须检查：

- 静态第一帧已经布局正确，动画不是用来修复错位。
- 每个移动对象都有稳定支点或路径；没有"漂浮"、"瞬移"、"穿过文字"。
- 每个移动对象都有可见轨道、清晰标签和必要性；如果删掉动画不影响理解，就删掉它。
- 箭头、标签、图例与主图形分层清楚，没有互相遮挡。
- 动画速度服务理解：不能快到看不清，也不能慢到像卡住。
- 图注明确说出这张图在讲什么，以及类比的边界。

### SVG 布局安全规则

生成 SVG 前先规划坐标，不要边写边挤：

1. **先分区**：把 viewBox 明确切成左/中/右或上/下区域，例如环形图 `0-300`、图例 `320-540`、关键数字 `560-700`。
2. **留安全边距**：所有文字和图形距离 viewBox 边缘至少 24px；长中文说明优先换行，不让文字贴边或溢出。
3. **图形和文字不要共享同一坐标带**：环形图、图例、大数字之间至少留 32px 空隙。
4. **长标签必须估算宽度**：中文按约 `font-size`，英文数字按约 `0.6 * font-size` 估算；放不下就缩短、换行或移到图注。
5. **不要依赖动画后的视觉位置**：静态初始布局必须已经正确；动画只能淡入、描线、强调，不负责把错位元素搬回正确位置。
6. **复杂图表优先增高 viewBox**：如果横向空间不足，把 `viewBox` 高度加到 300-420，用上下分区，不要把图例和数据硬塞到同一行。
7. **复杂动画优先增大画布**：涉及运动轨迹、波形叠加、时间阶段、物理类比时，优先用 `900x560` 或更大 viewBox，并把运动区和解释区分开。

### SVG 信息架构规则

先决定图的语义结构，再画视觉元素：

- **对比关系**：优先用并列卡片、表格或坐标轴。只有在表达严格的集合交集时才用维恩图；不要用重叠圆强行表示“有点像/都能用”，这会让交集标签位置和含义都变含糊。
- **流程/链条**：节点、连接线、连接标签必须分层。连接线走独立通道，最短可见线段不少于 48px；不要把“链接/升级/下一步”文字压在箭头或短线中间。需要解释连接含义时，用线下方的说明条或图注。
- **流程强调线 / 动画圆点失败案例**：不要把虚线 flow-line、粒子圆点、流动高亮线画在节点内部或文字基线上，即使它只是“强调路径”。典型错误是把横向虚线路径放在卡片中线，穿过“用户需求 / GPU / 数据中心”等副标题。正确做法是预留独立 flow channel，例如放在卡片下方 32-48px 的空白带，或放在卡片外侧连接通道；动画圆点必须沿同一独立通道运动，不能压到箭头、标签或卡片文字。
- **时间线**：日期和事件说明必须在不同基线，基线距离不少于 22px；上方事件统一“日期在上、说明在下”，下方事件统一“日期在上、说明在下”。竖线只连接节点到标签区边缘，不穿过文字。
- **中心辐射图**：连接线只能连接到卡片边缘，不进入文字区；卡片内部标题和说明基线距离不少于 20px。
- **数据图**：大数字和说明文字基线距离不少于 28px；右侧指标组如果超过 3 项，改成上下布局或增加 viewBox 高度，不要压缩行距。
- **标签背景**：当文字靠近线、圆、箭头或复杂形状时，给标签加浅色底/留白，避免文字和线抢视觉层级。
- **图标与 emoji**：不要依赖彩色 emoji 承担结构信息；emoji 只作辅助，关键含义必须由文字和形状表达。

### SVG 生成前草图

生成每个 SVG 前，先在心里形成一个坐标草图，并让代码体现这个草图：

```html

  viewBox 720 x 320
  left card: x 40-230
  middle card: x 265-455
  right card: x 490-680
  label band: y 235-275

```

如果一个 SVG 无法用 4-6 行说明清楚分区，就说明它太复杂，应拆成两个图或改用更简单的图型。

### SVG 字号规范

SVG 内部文字在 viewBox 缩放时会等比例缩小。必须使用足够大的字号保证可读性：

| 角色 | 最小字号 | 说明 |
|---|---|---|
| 图表主标题 | 18px | viewBox 内的标题文字 |
| 数据标签/节点标题 | 16px | 柱子顶数字、节点名 |
| 轴标签/图例 | 14px | 坐标轴、图例说明文字 |
| 节点描述/注释 | 13px | 辅助说明文字 |
| 大数字（仪表盘） | 30px+ | 核心数值展示 |
| 禁止使用 | ≤11px | 一律不可用，缩到屏幕后会消失 |

### SVG 交互规范

- **悬停高亮**：鼠标移到某个节点上时，该节点高亮并显示说明文字
- **点击展开**：点击某个部分时，展开更详细的解释
- **步骤推进**：对于流程类概念，提供"下一步"按钮逐步演示
- 使用纯 CSS + 少量 inline JS 实现，不引入外部依赖

## Step 5 · 填充内容

### 语言风格

- **开头**：直接说"XX是……"，不要"今天我来给大家介绍……"
- **类比**：选一个就够，不要堆砌多个类比
- **句子长度**：一句话不超过 30 个字，复杂概念拆成多句
- **段落长度**：一段不超过 4 行（屏幕显示）
- **术语处理**：首次出现时括号解释，如"API（一种让不同软件对话的接口）"
- **语气**：像一个耐心的朋友在聊天，不像老师在讲课
- **少写串场**：不要写"下一节要讲什么"、"现在你能理解..."这类提示。用标题、图解、结论句完成过渡。
- **文章主线**：复杂概念要像一篇科普文章，有开篇路线、章节之间的因果衔接、图后的回扣段，不要像一组独立卡片。

### 可视化优先抽象

复杂概念页面不是把文字排漂亮，而是把知识转成可观察的结构、状态和变化。每个关键段落都先判断能不能可视化；能可视化就优先做成 SVG/CSS 动画或交互。

生成每节前做一次"可视化映射"：

```text
知识句：普通逻辑门把 0 改成 1，把 1 改成 0
生活场景：白牌/蓝牌进入翻牌机
抽象对象：两种状态、改变规则、输入、输出
变化规则：白 -> 蓝，蓝 -> 白；命名后可写成 NOT(x)=1-x
视觉映射：白/蓝牌是两种状态；翻牌机是中间机器；动画展示输入经过机器后换色
术语命名：白/蓝 -> 0/1 bit；翻牌机 -> gate；换色规则 -> NOT
交互/动画：白牌进入机器，输出蓝牌；第二轮蓝牌进入，输出白牌
```

优先选择这些可视化形式：

- **状态变化**：状态机、开关、仪表、前后对比、逐帧变换
- **流程因果**：管道、流水线、箭头路径、分支、反馈循环
- **数学规则**：公式 + 输入输出示例 + 动态代入
- **物理直觉**：波、摆、力、能量、轨迹，但必须说明类比边界
- **结构关系**：层级图、组件图、依赖图、系统边界
- **规模变化**：柱状图、增长曲线、数量级尺、热力图
- **不确定性**：概率分布、抽样、亮度/振幅变化、测量前后对比

文字只承担三件事：给结论、解释图中看不到的边界、消除误解。不要让关键机制只停留在段落里。

### 列表与步骤组件规范

步骤列表要服务阅读节奏，不要制造视觉断裂。

- 编号优先用小圆形、小胶囊、细竖线时间轴或左侧窄编号列，不要用大色块占据整列。
- 编号容器建议 `32-40px` 高，宽度只包住数字或短标签；禁止用 `80px+` 的深色块作为普通步骤编号。
- 文本应是视觉主体：标题和说明靠近编号，编号不能比正文更抢眼。
- 步骤卡片可以用浅底和细边框；深色只用于小编号、当前步骤或关键强调，不要在每个步骤都铺大面积深色。
- 移动端仍保持编号和文字同一行起步，除非空间不足；不要让编号单独占一整行导致阅读断裂。
- 若步骤超过 4 个，优先使用时间轴、分组卡片或紧凑列表，而不是重复大卡片。

### SVG 内容填充

每个 SVG 必须：
1. 有明确的标题（`<text>` 或旁边的 `<p>`）
2. 有图注说明"这张图在讲什么"
3. 配色严格使用 Kami 设计 token
4. 文字大小 ≥ 13px（SVG 内绝对字号），禁止使用 ≤11px
5. 动画服务于教学目的
6. **图表占满整行**：使用 `.figure` 容器包裹 SVG，`display:block;width:100%;clear:both`，不与文字并排。图注在 SVG 下方居中
7. **无错位风险**：带 SVG `transform` 属性的元素不能套用会修改 CSS `transform` 的动画；生成后检查图例、大数字、标签没有压到主图形上
8. **语义图型正确**：对比不用伪维恩图，流程不用短箭头挤标签，时间线不用让事件名和日期共用同一基线
9. **专业动画要有设计意图注释**：包含 teaching claim、visual mapping、sequence、boundary
10. **类比动画要说明边界**：图注必须说明该类比解释什么、不解释什么
11. **颜色对比必须正确**：深色背景上的文字必须使用 `#faf9f5`、白色或同等亮度浅色；浅色背景上的文字使用 `#141413` 或 `#504e49`。不要依赖 SVG `fill` 属性覆盖 class，必要时用专用类或 inline style 确认最终渲染颜色。

- [ ] SVG 图解能独立传达信息（不依赖正文）
- [ ] 深色形状上的文字是白色/米白色，浅色形状上的文字是深色；浏览器最终渲染后对比度可读
- [ ] 动画能帮助理解，不是纯装饰
- [ ] SVG 动画没有覆盖元素原本的 `translate` / `rotate` / `scale`
- [ ] SVG 内图例、标签、大数字与主图形之间没有重叠
- [ ] SVG 连接线没有穿过文字，连接标签没有压在线上或箭头上
- [ ] 复杂 SVG 动画有清晰时序，不是所有元素同时循环

- [ ] 波形、叠加、因果变化等图解把输入、过程、输出分区展示，读者不需要猜线条含义
- [ ] 类比图有边界说明，避免把类比误认为真实机制
- [ ] 时间线的日期和事件说明分层清楚，没有上下文字互相压住
- [ ] 对比图选择了正确图型：并列比较用卡片/表格，真实集合交集才用维恩图
